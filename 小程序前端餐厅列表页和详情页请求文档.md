# 小程序前端餐厅列表页和详情页请求文档

## 1. 目标

本文档用于说明微信小程序中“餐厅列表页”和“餐厅详情页”如何正确请求当前 Django 后端。

当前后端局域网地址：

`http://192.168.170.105:8000`

当前业务接口前缀：

`http://192.168.170.105:8000/api/v1`

---

## 2. 总体原则

餐厅详情页不能自己拼商家 ID，也不能使用历史写死的商家 ID。

正确流程必须是：

1. 列表页先调用“餐厅搜索接口”
2. 从搜索结果中拿到商家 `id`
3. 跳转详情页时把这个 `id` 传过去
4. 详情页再用这个 `id` 请求“餐厅详情接口”

如果不按这个顺序做，就容易出现后端 `404`。

---

## 3. 餐厅列表页请求说明

### 3.1 接口地址

`POST /api/v1/diet/search/`

完整地址：

`http://192.168.170.105:8000/api/v1/diet/search/`

### 3.2 使用场景

用于获取当前用户附近餐厅列表。

### 3.3 请求方式

`POST`

### 3.4 认证要求

需要登录态，请在请求头中携带：

`Authorization: Bearer <token>`

### 3.5 请求参数

请求体中必须包含：

- `mode`
  固定传 `restaurant`
- `lng`
  用户当前位置经度
- `lat`
  用户当前位置纬度

### 3.6 请求参数说明

- `lng` 和 `lat` 必须是真实定位坐标
- 不要传空字符串
- 不要把经纬度写死成测试值后忘记切回

### 3.7 返回结果重点字段

后端返回结构中关注：

- `code`
  是否成功
- `data.recommendations`
  餐厅列表数组

每个餐厅对象重点字段：

- `id`
  商家唯一标识，后续详情页必须使用它
- `name`
  商家名称
- `address`
  商家地址
- `rating`
  商家评分
- `image`
  商家首图
- `estimated_calories`
  预估热量
- `health_light`
  红黄绿健康灯
- `health_score`
  健康评分

### 3.8 列表页处理要求

列表页拿到结果后，必须把每个餐厅的 `id` 保留下来。

页面跳转到详情页时，传递的应当是搜索结果里的 `id`，而不是餐厅名称、索引位置或本地生成值。

---

## 4. 餐厅详情页请求说明

### 4.1 接口地址

`GET /api/v1/diet/restaurant/{id}/`

完整格式：

`http://192.168.170.105:8000/api/v1/diet/restaurant/{id}/`

例如：

`http://192.168.170.105:8000/api/v1/diet/restaurant/B0M6PMKKKU/`

### 4.2 使用场景

用于获取单个餐厅的详细信息。

### 4.3 请求方式

`GET`

### 4.4 认证要求

需要登录态，请在请求头中携带：

`Authorization: Bearer <token>`

### 4.5 路径参数

- `id`
  必须来自餐厅搜索接口返回的 `recommendations[*].id`

### 4.6 返回结果重点字段

后端返回结构中关注：

- `code`
  是否成功
- `data.id`
  商家 ID
- `data.name`
  商家名称
- `data.address`
  商家地址
- `data.rating`
  商家评分
- `data.photos`
  商家图片数组
- `data.menu_items`
  菜单数组，当前可能为空数组

### 4.7 详情页处理要求

- 页面加载时先读取上一个页面传来的商家 `id`
- 用这个 `id` 请求详情接口
- 不要自己替换、截断、重组这个 `id`

---

## 5. 页面跳转要求

从列表页跳到详情页时，必须传递以下最关键字段：

- `id`

如果需要减少详情页首屏等待，也可以额外传：

- `name`
- `image`
- `rating`

但这些都只是展示优化字段，真正请求详情接口时仍然必须使用 `id`。

---

## 6. 常见错误与处理方式

### 6.1 错误一：详情接口 404

表现：

后端日志出现：

`GET /api/v1/diet/restaurant/某个ID/ 404`

原因通常是：

- 前端写死了旧商家 ID
- 前端使用了历史缓存 ID
- 前端没有从当前搜索结果里取 `id`

正确处理：

- 返回列表页重新搜索
- 使用本次搜索结果里的最新 `id`

### 6.2 错误二：列表为空

原因通常是：

- 经纬度没传对
- 小程序未授权定位
- 后端附近商家缓存为空，且高德接口未返回数据

正确处理：

- 先检查小程序定位权限
- 确认 `lng`、`lat` 为真实数值
- 确认后端高德接口配置有效

### 6.3 错误三：网络请求失败

原因通常是：

- 小程序请求地址仍是 `localhost`
- 不是同一局域网
- 后端没启动
- 后端电脑防火墙拦截 `8000`

正确处理：

- 基础地址统一改成 `http://192.168.170.105:8000`
- 确认后端运行在 `0.0.0.0:8000`
- 确认前后端电脑仍连接同一热点

---

## 7. 联调检查顺序

建议前端联调时按以下顺序检查：

1. 浏览器访问 `http://192.168.170.105:8000/api/docs/`
2. 确认后端服务可访问
3. 小程序先调用餐厅搜索接口
4. 检查返回列表里是否拿到了 `id`
5. 使用该 `id` 再请求餐厅详情接口
6. 若详情 404，优先检查 `id` 来源是否正确
