# 局域网与内网穿透联调文档

## 1. 目的

本文档用于说明当前项目在以下两种场景下如何联调：

1. 两台电脑在同一个局域网中通信
2. 两台电脑不在同一个局域网时，通过内网穿透访问后端

当前项目结构：

- 业务前端：微信小程序
- 后端：Django + Python
- 后台管理：独立 HTML 前端

---

## 2. 先区分两种联调方式

### 2.1 局域网联调

适用场景：

- 两台电脑连接同一个 Wi-Fi
- 两台电脑连接同一个手机热点
- 前端电脑和后端电脑能互相访问局域网 IP

特点：

- 配置最简单
- 速度快
- 不需要公网服务器

### 2.2 内网穿透联调

适用场景：

- 两台电脑不在同一个局域网
- 小程序或后台前端需要从外网访问本地 Django
- 想临时把本机开发环境暴露到公网做测试

特点：

- 可以跨网络访问本地服务
- 需要隧道工具或中转服务器
- 稳定性和安全性弱于正式部署

---

## 3. 局域网联调方式

### 3.1 什么时候算同一个局域网

以下情况通常都算同一个局域网：

- 同一个公司或家庭路由器
- 同一个手机热点

如果两台电脑都连到同一个手机热点，它们一般会被分配到同一网段，可以按局域网方式通信。

### 3.2 当前项目推荐做法

后端电脑启动 Django：

```powershell
cd C:\Users\zlj\Desktop\health_life_backend\health_life_backend_dj
.\start_lan.bat
```

或者手动启动：

```powershell
python manage.py runserver 0.0.0.0:8000
```

### 3.3 当前项目局域网地址

当前后端局域网地址示例：

`http://192.168.170.105:8000`

业务接口前缀：

`http://192.168.170.105:8000/api/v1`

后台管理接口前缀：

`http://192.168.170.105:8000/api/admin/v1`

### 3.4 局域网联调检查顺序

1. 后端电脑执行 `ipconfig`
2. 找到连接 Wi-Fi 或手机热点的 IPv4 地址
3. 前端电脑先浏览器访问：

`http://后端电脑IP:8000/api/docs/`

4. 如果能打开，说明局域网链路正常
5. 再让小程序或后台管理前端请求这个 IP

### 3.5 常见问题

#### 问题一：前端请求的是 `127.0.0.1`

含义：

`127.0.0.1` 指的是“前端电脑自己”，不是后端电脑。

正确做法：

必须改成后端电脑的局域网 IP。

#### 问题二：浏览器打不开后端地址

优先排查：

- 后端是否监听在 `0.0.0.0:8000`
- 两台电脑是否仍连接同一热点或同一路由器
- Windows 防火墙是否放行 `8000`

#### 问题三：后台管理跨域失败

当前项目已经改为：

- 后台管理前端请求局域网后端地址
- `/api/admin/` 路径放宽 CSRF，适配 JWT 管理接口

## 4. 为什么还需要内网穿透

局域网联调只能在“同一个网络环境”中工作。

如果出现以下情况，就需要内网穿透：

- 前端电脑和后端电脑不在同一个热点或路由器
- 需要让外地设备访问你本机的 Django
- 需要让别人临时访问你的本地开发接口

---

## 5. 当前项目已有的内网穿透方式

项目里已经有一个脚本：

`C:\Users\zlj\Desktop\health_life_backend\health_life_backend_dj\start_tunnel.bat`

它的作用是：

- 把本机 `127.0.0.1:8000`
- 通过 SSH 反向隧道映射到远端服务器 `8081` 端口

也就是说，外部可以通过远端机器的 `8081` 端口访问你本机 Django。

---

## 6. 内网穿透的基本原理

### 6.1 你的本机

本地 Django 服务运行在：

`127.0.0.1:8000`

### 6.2 隧道脚本

通过 SSH 建立反向隧道，把本地 8000 映射到远端服务器的 8081。

### 6.3 外部访问方式

外部访问：

`http://远端服务器IP:8081`

就等价于访问你本机的 Django。

---

`http://47.95.215.220:8081/api/v1`

---

## 9. 当前项目使用内网穿透时的安全点

### 9.1 适合做什么

- 临时测试
- 联调
- 给前端或测试同学短期访问

### 9.2 不适合做什么

- 长期正式生产环境
- 高并发访问
- 对安全和稳定性要求高的线上场景

### 9.3 原因

- 本机断网或休眠，服务立刻不可用
- 隧道中断会导致访问失效
- 安全边界较弱

---

## 10. 推荐的联调策略

### 10.1 优先顺序

推荐按这个顺序选择：

1. 同热点或同 Wi-Fi：优先局域网联调
2. 不同网络但只是临时测试：使用内网穿透
3. 稳定长期对接：部署到正式云服务器

### 10.2 当前项目建议

如果你和前端电脑能连接同一个手机热点，就优先使用局域网联调，不需要内网穿透。

只有在“不在同一网络”时，再切到 `start_tunnel.bat` 方案。

---

## 11. 排障清单

### 11.1 局域网不通

依次检查：

1. Django 是否启动
2. 是否监听 `0.0.0.0:8000`
3. 前端请求地址是否仍是 `127.0.0.1`
4. 防火墙是否放行 `8000`
5. 两台电脑是否仍在同一个局域网

### 11.2 内网穿透不通

依次检查：

1. 本地 Django 是否启动
2. `start_tunnel.bat` 是否持续运行
3. 远端服务器 `8081` 是否开放
4. 当前访问地址是否使用了远端 IP 和 `8081`
5. 隧道是否中断

---

## 12. 一句话结论

如果前后端在同一个热点或同一个 Wi-Fi，下发局域网 IP 就够了，不需要穿透。

如果不在同一个网络，就先启动 Django，再运行 `start_tunnel.bat`，然后让前端访问远端服务器的穿透地址。
