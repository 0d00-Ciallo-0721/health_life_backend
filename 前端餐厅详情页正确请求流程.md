# 前端餐厅详情页正确请求流程

## 1. 正确链路

餐厅详情页不能写死商家 ID，也不能复用旧缓存里的 ID。

正确顺序必须是：

1. 先调用餐厅搜索接口
2. 从搜索结果列表里取某一项的 `id`
3. 再用这个 `id` 请求餐厅详情接口

---

## 2. 餐厅搜索接口

请求：

```http
POST /api/v1/diet/search/
Authorization: Bearer <token>
Content-Type: application/json
```

请求体示例：

```json
{
  "mode": "restaurant",
  "lng": 116.411136,
  "lat": 39.911048
}
```

返回结果里的 `data.recommendations` 数组，每一项都会带商家 `id`：

```json
{
  "code": 200,
  "data": {
    "recommendations": [
      {
        "id": "B0xxx",
        "name": "某餐厅",
        "address": "某地址",
        "rating": 4.6,
        "image": "https://...",
        "estimated_calories": 600,
        "health_light": "yellow",
        "health_score": 70
      }
    ]
  }
}
```

前端详情页必须使用这里返回的 `id`。

---

## 3. 餐厅详情接口

请求：

```http
GET /api/v1/diet/restaurant/{id}/
Authorization: Bearer <token>
```

示例：

```http
GET /api/v1/diet/restaurant/B0xxx/
```

返回：

```json
{
  "code": 200,
  "data": {
    "id": "B0xxx",
    "name": "某餐厅",
    "address": "某地址",
    "rating": 4.6,
    "photos": ["https://..."],
    "menu_items": []
  }
}
```

---

## 4. 前端必须避免的错误

- 不要写死 `B0JD371VD7` 这种历史 ID
- 不要在详情页本地拼一个假的商家 ID
- 不要在换坐标后继续用上一次搜索结果的旧 ID

---

## 5. 推荐前端实现

列表页跳转详情时，把搜索结果中的 `id` 一起带过去：

```js
navigateTo({
  url: `/pages/restaurant/detail?id=${item.id}`
})
```

详情页加载时再请求：

```js
GET /api/v1/diet/restaurant/${id}/
```

如果接口返回 404，说明这个 `id` 当前不在后端缓存中，应该返回列表重新搜索，而不是继续重试旧 ID。
